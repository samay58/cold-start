# Firefox Port Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Phase 0 is a gate; do not start Phase 1 until its exit criteria are recorded in this file.

**Goal:** Ship a Mozilla-signed, self-distributed Firefox build of the Cold Start extension while keeping Chrome behavior unchanged, hardening extension auth to a two-browser identity model, and adding a Firefox lane to CI.

**Product verdict:** Feasible, and smaller than a generic side-panel port. Firefox cannot reproduce Chrome's per-tab side panel; its sidebar is one visible surface per browser window, shared across every tab in that window. Cold Start survives this better than most side-panel products because the panel already keys off the domain captured at the last toolbar click (`storage.session.activeDomain` written in `background.ts`, consumed live via `storage.onChanged` in `sidepanel.tsx`), not off the current tab. The honest Firefox framing: the sidebar shows your last researched company in this window; click the toolbar button on any tab to research that site. Samay accepted this model on 2026-07-13 (see resolved decisions), so the port proceeds; there is no API workaround and none is needed.

## Verification Trail (2026-07-13)

Every load-bearing claim below was checked against primary sources on 2026-07-13 (MDN, extensionworkshop.com, Bugzilla, npm, project source), not carried over from secondhand research. Constraints each source imposes:

- MDN `background` manifest key: Firefox MV3 has no `background.service_worker`; it runs a non-persistent event page from `background.scripts`. Dual-key manifests work on Firefox 121+ / Chrome 121+, but per-browser manifests are cleaner and easier for AMO review. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/background
- MDN Sidebars + `sidebarAction`: sidebar visibility is window-scoped; `sidebarAction.open()` takes no parameters, opens in the active window, and must be called synchronously inside a user-action handler (an `await` before it loses the gesture). `sidebarAction.toggle()` exists under the same rules. Each window gets its own instance of the sidebar document. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/user_interface/Sidebars and /API/sidebarAction/open and /User_actions
- MDN `sidebar_action` manifest key: `open_at_install` defaults to `true`; the Firefox build must set it to `false`. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/sidebar_action
- MDN `tabs` API: `tabs.onActivated` fires without any permission, but its payload is only `tabId`/`windowId`/`previousTabId`. Reading a tab's `url`/`title`/`favIconUrl` requires the `tabs` permission, matching host permissions, or a live activeTab grant. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/API/tabs
- MDN `permissions` (activeTab): qualifying gestures include the toolbar action click and "clicking a button on a page bundled with the extension". Whether an in-sidebar button click grants activeTab for the content tab (rather than the extension page) is undocumented; Phase 0 tests it empirically. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/permissions
- MDN Chrome incompatibilities: Firefox supports the `chrome.*` namespace with callback-style APIs in MV3 as a porting aid. Cold Start's storage/runtime/action callback code runs unmodified. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/Chrome_incompatibilities
- MDN `runtime.id` + `browser_specific_settings`: when `gecko.id` is set, `runtime.id` returns that stable developer-chosen ID, not the per-install UUID. The existing `x-cold-start-extension-id` header therefore carries a stable Firefox identity with zero extension code change. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/API/runtime/id
- Bugzilla 1405971 (open, unresolved): Firefox currently sends `Origin: moz-extension://<per-install-random-UUID>` on extension fetches; a fix was backed out. Server logic must not depend on any particular Firefox Origin value. https://bugzilla.mozilla.org/show_bug.cgi?id=1405971
- Chrome favicon docs + Bugzilla 1315616: the `favicon` permission and `_favicon/` URL are Chrome-only; Firefox has no equivalent (open feature request). https://developer.chrome.com/docs/extensions/how-to/ui/favicons
- MDN `storage.session`: Firefox 115+, memory-only, cleared on browser close, 10 MB quota. Current usage fits. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/API/storage/session
- MDN `host_permissions`: from Firefox 127, MV3 host permissions appear in the install prompt and users can revoke them afterwards; the panel must show a useful error when the API origin fetch fails. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/host_permissions
- MDN `incognito`: `"not_allowed"` (Firefox 67+) makes private windows invisible to the extension; Firefox treats `"split"` as `not_allowed`. https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/incognito
- Extension Workshop, built-in data consent: extensions first submitted after 2025-11-03 must declare `browser_specific_settings.gecko.data_collection_permissions`. Transmitting the active tab's domain is `browsingActivity`. The built-in consent UI exists in Firefox desktop 140+; setting `strict_min_version` to 140.0 is Mozilla's own recommended way to rely on it exclusively. https://extensionworkshop.com/documentation/develop/firefox-builtin-data-consent/
- Extension Workshop, signing and self-distribution: release/beta Firefox require Mozilla signing even for unlisted builds; automated signing can take up to 24 hours and any version can be pulled into manual review; unlisted extensions handle their own update distribution (`update_url` or manual). https://extensionworkshop.com/documentation/publish/signing-and-distribution-overview/
- Extension Workshop, source code submission: bundled/minified output requires uploading matching source plus build instructions for every version; reviewers rebuild and diff, expecting no differences. Default reviewer environment: Ubuntu 24.04.4 LTS ARM64, Node 24.14.0, npm 11.9.0 (used when the developer does not specify one). Minification allowed, obfuscation prohibited. https://extensionworkshop.com/documentation/publish/source-code-submission/
- Extension Workshop, add-on policies (updated 2025-06-23): browsing-activity transmission is permitted only as part of the primary function; passive or continuous background transmission requires explicit consent; data from private browsing must not be stored; remote executable code prohibited; encrypted transport required. AMO no longer requires a hosted privacy policy; a self-hosted one is encouraged (Cold Start already serves `/privacy`). https://extensionworkshop.com/documentation/publish/add-on-policies/
- CRXJS `@crxjs/vite-plugin` 2.7.1 (npm, 2026-07-01): ships `browser: 'firefox' | 'chrome'`; `plugin-background.ts` emits `background.scripts` for Firefox; dev-server CORS accepts `moz-extension://` origins. It does NOT translate `side_panel` to `sidebar_action` (zero occurrences in the repo); we own that manifest delta. Maintenance is active again (new maintainer team, ~6 releases over the last 9 months, Vite 8 support in June 2026). https://github.com/crxjs/chrome-extension-tools
- WXT 0.20.27 (pre-1.0 RC line): auto-maps a side-panel entrypoint to both browsers, per-browser manifests, `wxt zip --sources`. Verified capable; not adopted (decision below). https://wxt.dev
- Playwright: extensions load only in Chromium persistent contexts; no Firefox extension loading. The Firefox smoke lane uses web-ext and Selenium/geckodriver (`install_addon(..., temporary=True)`). https://playwright.dev/docs/chrome-extensions
- web-ext 10.5.0: `lint`, `run`, `build`, and `sign --channel=unlisted` (AMO JWT credentials) all current. https://extensionworkshop.com/documentation/develop/web-ext-command-reference/
- Firefox dev-server caveat: extension CSP blocked `http://localhost` script sources until Firefox 147 for temporarily loaded extensions, so CRXJS HMR against older Firefox does not work. Development uses `vite build --watch` plus `web-ext run`; HMR is a nice-to-have, not a blocker.

Research provenance: an external deep-research pass (ChatGPT) produced the candidate claims; everything above was independently re-verified. Two of its claims died in verification: "AMO requires a privacy policy whenever data leaves the device" (rescinded by Mozilla 2025-06-23) and its proposed tab-follow UX, which silently required the `tabs` permission it never flagged.

## What Actually Changes (repo ground truth)

Only seven files in `apps/extension/src` touch `chrome.*`: `background.ts`, `card-cache.ts`, `CompanyLogo.tsx`, `ResearchLayerPanel.tsx`, `sidepanel-network.ts`, `sidepanel.tsx`, `theme.ts`. Because Firefox supports `chrome.*` callbacks in MV3, all storage, runtime-id, and action listener code ports as-is. The genuinely Chrome-only surface is:

- `chrome.sidePanel.setPanelBehavior` and `chrome.sidePanel.open` (`background.ts`, two call sites)
- `chrome.runtime.getURL("_favicon/...")` (`clipping-model.ts:127`, one call site)
- the manifest: `side_panel`, the `sidePanel` and `favicon` permissions, `background.service_worker`

One latent bug the port must fix rather than inherit: the favicon helper guards on `runtime?.getURL` existing. On Firefox that check passes and produces a `moz-extension://<uuid>/_favicon/...` URL that 404s. The gate must become a manifest-permission check.

Server side, `assertExtensionRequest` (`apps/web/src/lib/extension-auth.ts:81`) currently accepts extension-ID match OR allowed-origin match, then validates the bearer token. Since Firefox origins are per-install random UUIDs, origin allowlisting cannot identify Firefox clients, and the OR shape lets an origin substitute for identity. This hardens to: ID required, token required, origin demoted to a Chrome-only consistency check. No request or response shape changes, so per repo convention no API contract version bump is needed; the server change is a strict-superset deploy that must go out before any Firefox build is distributed.

## Decisions

**Tooling: stay on CRXJS, upgrade to 2.7.x. Do not migrate to WXT now.** The external research recommended WXT, but its strongest argument ("CRXJS is Chrome-oriented and shaky") did not survive verification: CRXJS 2.7.1 has a first-class Firefox target and active maintenance. What WXT would add (automatic `sidebar_action` mapping, source-zip helpers) is one to two days of owned scripting on a manifest that is already a plain function. What WXT would cost is restructuring entrypoints, both Vite configs, and the tuned Playwright smoke and shim harness (two to four days) on a pre-1.0 framework. For a solo-maintained repo where the check gate is the safety net, the smaller diff wins. Revisit trigger: WXT ships 1.0 and either a third browser target appears or CRXJS regresses.

**UX: last-click-wins plus a stale-tab hint, no `tabs` permission.** Semantics match Chrome exactly: the toolbar click captures the domain and opens the panel; the panel never follows tabs on its own. The one Firefox difference is that the sidebar stays visible after a tab switch, so it can show a company that no longer matches the current tab. The fix is a quiet staleness hint driven by `tabs.onActivated` (fires with no permission, carries only tab and window IDs): background stores `lastResearchTabId`/`lastResearchWindowId` at click time; when the sidebar's window activates a different tab, show "Different tab. Click the Cold Start button in the toolbar to research it." Never read the new tab's URL, never auto-research. This keeps the AMO data story minimal: domains are transmitted only on a deliberate toolbar click, which is squarely the built-in-consent primary-function case. The rejected alternative (auto-follow with domain display) requires the `tabs` permission and turns tab-switching into passive browsing-activity transmission.

**Auth: extension ID required, token required, origin demoted.** Production accepts a request only when the ID header matches an entry in the allowed-ID list (Chrome store ID or Firefox gecko ID) AND the bearer token matches. If an Origin header is present and starts with `chrome-extension://`, it must exactly match the configured Chrome origin; `moz-extension://` and absent origins are ignored as identity (bug 1405971 makes them unreliable and unenumerable). `http(s)` origins stay rejected in production unless explicitly allowlisted, exactly as today. Per-browser bearer tokens so one can be revoked independently. Honest posture, unchanged from `SECURITY.md`: the token is the secret; an ID header is caller-controlled routing metadata, and a token embedded in a public XPI is extractable. That is acceptable for an unlisted beta with rate limits and rotation; a public AMO listing upgrades this to user-bound tokens (Phase 3, out of scope).

**Distribution: unlisted (self-distributed) first.** Signed XPI handed directly to known users; still fully subject to validation, policy, and possible manual review. Listing on AMO is a separate later decision.

**Floor: Firefox 140 (`strict_min_version: "140.0"`).** Lets the built-in data-consent UI be the only consent surface. Older Firefox simply cannot install the extension, which Mozilla documents as the intended option for new extensions.

**Decisions resolved with Samay, 2026-07-13:**

1. Gecko ID: `cold-start@semitechie.vc`. Chosen once, it can never change without becoming a different extension to Firefox.
2. Toolbar click while the sidebar is already open: `sidebarAction.open()` no-op (Chrome parity).
3. Consent and privacy copy approved in shape: "Research will be saved at a public URL." The word "public" is load-bearing; a reviewer installs the extension, sees the card publish at `/c/{slug}`, and checks the disclosure matches. Final copy still gets a read before AMO submission.
4. Window-scoped sidebar accepted, and arguably preferred for this product: Chrome's per-tab panel disappears when the user tabs away mid-generation, while the Firefox sidebar keeps the research on screen as a reference surface across the company's site, docs, and LinkedIn tabs. The Phase 0 walkthrough is confirmation, not a go/no-go gate.

## Phase 3: Public Listing Hardening (out of scope, recorded so nobody trips into it)

A public AMO listing makes the embedded shared token extractable at scale. That milestone requires user-bound short-lived tokens (login through the web app), quotas, and abuse monitoring. Do not list publicly on the shared-token model.

## Risks

1. Window-scoped sidebar UX: accepted by Samay 2026-07-13 (sticky research surface reads as a feature for this product, not a loss). Residual risk is only the small stuff the stale-tab hint handles; the Phase 0 walkthrough confirms rather than gates.
2. AMO review friction: reproducible-build drift or reviewer questions about publishing researched cards at public URLs. Mitigation: reproducibility CI job, plain-language consent copy approved up front, unlisted channel first.
3. CRXJS Firefox-target edge cases (it is newer than the Chrome path). Mitigation: the spike exercises build, dev-run, and smoke early; WXT remains the documented fallback with a two-to-four-day switching cost.
