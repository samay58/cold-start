LESSON 0001 · COLD START

Follow one company generation from request to saved card

This is the first path to learn. A user asks for a company. Cold Start decides whether to run, fetches evidence, builds a cited card, saves it, and leaves enough trace data for us to understand what happened.

Why this path matters

Cold Start is not trying to write a nice company summary from memory. It is trying to build a useful first read from sources the product can point back to.

That shapes the code. The app has to answer four questions in order:

  1. Do we need to run, or do we already have a fresh card?
  2. Which sources are good enough to use for this company?
  3. Which facts can we extract with citations?
  4. What should be saved, shown publicly, or kept behind the extension?
The useful mental model: generation is a job with receipts. The card is the output. The trace explains how that output was made.

The whole path

1. Request POST /api/generate parses the domain, mode, optional section, confirmation flag, and extension auth.
2. Cache and active run check The route checks whether a usable card already exists or whether a run is already queued or running.
3. Queue If work is needed, the route writes a queued row in generation_runs, records a user-visible event, and sends card/generate.requested to Inngest.
4. Worker generateCardFunction marks the run as running, starts the trace, fetches sources, calls the card pipeline, and records progress.
5. Card build The pipeline turns accepted sources into a typed ColdStartCard. Facts without citations do not get to pretend they are known.
6. Save and read The database stores the full card JSON. Public reads remove synthesis. The extension can read the fuller card after auth.

Where to look

Question First file What it tells you
Was the run queued, cached, blocked, or already active? apps/web/src/app/api/generate/route.ts The request policy. This file decides whether the app starts work.
What happened during the slow part? apps/web/src/inngest/functions.ts The worker path. It owns run state, events, source fetching, pipeline calls, storage, and failure handling.
Where do provider calls and source filtering happen? apps/web/src/inngest/source-fetching.ts Direct Exa, StableEnrich, provider budget traces, source merging, and the source gate.
How does source text become a card? packages/pipeline/src/generate-card.ts The card build. It assembles facts, citations, synthesis, verifier results, and trace patches.
What shape is a valid card? packages/core/src/card.ts The schema. This is where citation-bearing facts, synthesis, signals, team, funding, and validation rules meet.
Where are runs, traces, cards, and events stored? packages/db/src/repository.ts The compatibility export for the DB layer. The actual functions now live under packages/db/src/repositories/.

What the trace gives us

Your instinct about the runs is right. We do not have to guess what happened after generate-card. The app stores run state and timing data.

The important pieces live in generation_runs.trace_json:

  • steps: named steps such as planning, source fetching, card generation, enrichment, and storage, with durations when available.
  • milestones: user-facing timing points such as first usable card, contacts ready, or analysis ready.
  • providers: source counts, failures, endpoint yield, wallet deltas, and estimated provider spend.
  • llm.calls: model, stage, provider, token counts, duration, status, and estimated cost for model calls.
  • sourceGate: accepted and rejected source counts, plus examples.
  • extraction and synthesis: how many citations survived, whether fallback was used, and whether synthesis was gated or verified.
  • failure: where a failed run broke and what message was recorded.

So the teaching path should use both sources: code for how the system is supposed to work, traces for what actually happened in a run.

Try it locally

If your local stack is running and your env has the needed keys, this queues a basics run:

curl -i -X POST http://localhost:3000/api/generate \
  -H 'content-type: application/json' \
  -d '{"domain":"cartesia.ai","confirmStart":true}'

The response should usually be one of three things:

  • cached: a usable stored card already exists.
  • queued: the route created a run and handed work to Inngest.
  • running: a run already exists, so the app returns the current state instead of starting a duplicate.

A real run can spend provider or LLM money. For learning, it is fine to read the path first and run commands later.

After this, ask the agent to walk the chosen file with you. The point is not to memorize file names. The point is to learn what each layer owns.

Answer key

Start with apps/web/src/app/api/generate/route.ts. A still-generating state is first a run-state question. The route tells you whether the request was cached, queued, running, blocked, or handed to Inngest.

If the route did queue a run, move to apps/web/src/inngest/functions.ts and the latest generation_runs.trace_json. That pair tells you what the worker tried, how long each step took, which providers returned useful data, and where the run failed or finished.
