LESSON 0002 · COLD START

The whole app in one map

Cold Start is easiest to understand if you stop seeing it as “a web app plus an extension” and start seeing it as one evidence object with several controlled doors into it.

One card.
Two surfaces. Public facts stay shareable. Investor judgment stays gated.

The purpose creates the architecture

The product is built for the first ten minutes of company research. A user gives Cold Start a company website. Cold Start returns a sourced company card that can be reopened, shared, cached, and extended.

The public page must be careful: facts, citations, source quality, and honest gaps. The extension can be sharper: investor synthesis, open questions, and richer research sections. That split is the load-bearing design choice described in SPEC.md and the operator-facing explanation in README.md.

The app map

apps/web The Next.js app. It owns public routes, APIs, generation queueing, extension auth, and the Inngest endpoint.
apps/extension The Chrome side panel. It reads the current tab, asks the API for cards and run status, then renders the private workbench.
packages/core The contract layer. Card shape, citations, public redaction, trust helpers, research-section taxonomy, and slug rules.
packages/db The storage layer. Drizzle schema, card reads and writes, run state, TTLs, evidence rows, sources, and research sections.
packages/providers The source-gathering layer. Direct Exa, StableEnrich, AgentCash, Firecrawl, Websets, EDGAR, and provider budgets.
packages/llm The language layer. Extraction, synthesis, verifier, research plans, research-section writing, model routing, and call traces.
packages/pipeline The assembly line. It turns raw sources into a typed card, merges provider facts, enriches weak blocks, and gates synthesis.
packages/ui The shared public card UI. It renders the catalogue card, citations, key values, public research, and source ledger.
eval and scripts The truth tools. Traces, golden evals, provider matrix checks, repair scripts, and cost or UX benchmarks.

The main story, in plain English

Request

POST /api/generate receives a domain. It normalizes the domain, decides whether this is basics, analysis, or a section job, checks auth, checks cache, and avoids duplicate active runs.

Queue

The route records a queued run and sends card/generate.requested to Inngest. The request handler does not do the expensive research itself.

Work

generateCardFunction becomes the run owner. It records product-facing events, gathers models and providers, and calls the pipeline.

Assemble

generate-card.ts fetches evidence, asks the LLM for structured sections, merges provider facts, runs block enrichment, validates citations, and creates one ColdStartCard.

Persist

repository.ts writes cards.card_json, generation run state, citations, claims, sources, and freshness windows.

Render

The public route reads a redacted public card. The extension route reads the full card only after extension auth. Both start from the same stored object.

Why the code is split this way

PUBLIC SURFACE

The public page needs to be shareable and safe. It gets sourced facts, citations, source mix, public research sections, and no private synthesis. This is why findPublicCardBySlug calls publicCard(...) before returning JSON.

EXTENSION SURFACE

The extension is the investor workbench. It can show synthesis and gated research because /api/extension/cards/{slug} first checks the extension headers and bearer token.

The contract to remember

packages/core/src/card.ts is the center of gravity. It says what a card is allowed to be. The most important rule is simple: if a resolved fact has a non-null value, it needs citation IDs, and those IDs must resolve to real top-level citations.

That rule is why Cold Start can prefer an honest thin card over a full-looking card full of guesses. Missing is allowed. Unsupported is not.
